## Verdict

**FIX-THEN-SHIP — three blockers in routing must land before further QA can be meaningful.**

The most important finding is a routing-level regression: the three persona landing pages
(`/dashboard`, `/portal`, `/reviewer/portal`) all return 404 against the running dev server even
with valid `site_gate` + `demo_user` cookies. Other routes inside the same route groups (`/companies`,
`/forms`, `/invoices`, `/payouts`, `/credentials`, `/reassignments`, `/prospects`) compile and serve
200, so the failure is concentrated on those three exact paths. This breaks every persona "home"
surface and propagates as a long tail of "missing keyword" / "not rendering" issues across other
scenarios — the moment the test driver lands at `/reviewers`, Next.js redirects it to
`/login?redirect_url=…` (because middleware sees the persona was bumped before the demo cookie was
re-honored on a 404 response), and from there nothing else can be exercised.

Once those three blockers are fixed, the rest of the issues fall into:
- 4 critical: server-side input-validation failures returning 500 instead of 4xx (multipart-only
  endpoints crash on JSON bodies — see issues 026, 027, 028, 029).
- 4 high: 3 protected endpoints accessible without auth (security), plus the active_cases vs DB
  drift / rendering issue.
- 20 medium / 3 low: UI copy and surface checks that are partly real and partly downstream of the
  blocker landing-page issue.

## Blocker Fix Plan

1. **Persona landing 404s (issues 001, 002, 003)** — `/dashboard`, `/portal`, `/reviewer/portal`
   return 404 in the running build despite their sibling routes resolving fine. Likely candidates:
   - Page-component or layout-component imports a missing/failed module that Next is silently
     converting to `_not-found` for the specific path. Check console for `Module not found` near
     the landing pages. The `/(dashboard)/dashboard/page.tsx` references `supabaseAdmin` from
     `@/lib/supabase/server` — if that module throws on import, dev server falls back to 404.
   - Same investigation for `app/(client)/portal/page.tsx` and `app/(dashboard)/reviewer/portal/page.tsx`.
   - Alternatively, a recently merged change to a `loading.tsx` / `error.tsx` boundary may be the cause.

2. **500s on JSON-body posts to multipart endpoints (issues 026, 027, 028, 029)** —
   `/api/providers/import`, `/api/upload/chart`, `/api/upload/form-template`, `/api/clients/onboard`
   crash with `TypeError: Content-Type was not one of "multipart/form-data" or "application/x-www-form-urlencoded"`.
   These should return 415 (Unsupported Media Type) or 400, not 500. Wrap the `request.formData()`
   call in try/catch and surface the proper status. (See dev log lines for stack traces.)

3. **Unauthenticated read access (issues 032, 033, 034)** — `GET /api/payouts`, `GET /api/reassignments`,
   `GET /api/notifications` return 200 with real data when called without a `demo_user` cookie. Add
   the same auth guard already present on `/api/clinics`.

4. **Notification body contains literal "undefined"** (`overall: undefined` in the
   `Hunter Health` feedback notification) — the notification template needs to default missing
   fields to a placeholder rather than letting `String(undefined)` leak into the body.

5. **DB integrity drifts** — orphan `review_cases` (NULL `company_id` / `provider_id`),
   reviewers with cases outside their `specialties[]`. These are seed-data issues but should be
   tightened with NOT NULL + FK constraints.
